# GitHub Pages publish verification — design

## Context

`GitHubPagesPublisher` (`audibleweb/publishers/github_pages.py`) is the
production publisher target per `docs/design.md` — `LocalPublisher` (already
verified live this session) is the dev-only fallback. Initial assumption
going into this brainstorm was that `GitHubPagesPublisher` had zero test
coverage. That assumption was wrong: `tests/test_publishers.py` already has
16 passing tests covering `publish`, `update_feed`, the atomic
`publish_and_update_feed`, clone reuse, no-op commits when nothing changed,
token redaction in error messages, episode rotation (`max_episodes`), and the
audio-size cap (`max_size_mb`) — all using a local bare git repo as a
stand-in remote, the same no-network-dependency technique used elsewhere in
this codebase's test suite.

The one real gap: every existing test passes `remote_url=str(bare_repo)`
explicitly, which bypasses the only line none of them exercise —
`self._remote_url = remote_url or f"https://{token}@github.com/{repo}.git"`
(`github_pages.py:49`). Production code (`core/pipeline.py`'s
`_build_publisher`) never overrides `remote_url`, so this is the actual code
path a real run takes and the only thing genuinely untested.

Beyond that, nothing has ever exercised this publisher against a **real**
GitHub remote — real network, real PAT auth, real GitHub Pages serving. That
class of gap is exactly where this session's real bugs were hiding on the
URL extraction path (trafilatura misuse) and the audio-route path (Flask
`root_path` resolution) — both invisible to mocked unit tests, both only
surfaced by a live run.

## Prerequisites (completed during brainstorming, not deferred)

- Test repo: `easilyBaffled/reader-rss-source` (was completely empty — zero
  branches). Bootstrapped during this session:
  - `main` branch: one commit (`README.md`), pushed.
  - `gh-pages` branch: orphan branch, one commit (placeholder `README.md`),
    pushed. `GitHubPagesPublisher._ensure_clone()` requires this branch to
    already exist on the remote (its own docstring: "Known limitation... not
    handled here") — without it, `git clone --depth 1 --branch gh-pages`
    fails outright.
  - Both pushes used ambient git credentials (macOS keychain) — no PAT was
    needed for the bootstrap itself.
- PAT: a real GitHub fine-grained PAT, scoped to just this repo with
  Contents: Read and write, is in `.env` as `GITHUB_PAT` (gitignored,
  confirmed via `git check-ignore -v .env`). Generated by the user directly
  in their browser; the raw value did land in this chat transcript despite
  using the `!` prefix (a harness limitation, not something either of us
  could avoid) — flagged to the user, low real risk since the token is
  scoped to a disposable test repo, their call whether to rotate it.

## Goal

Verify `GitHubPagesPublisher.publish_and_update_feed()` — the only method the
real pipeline calls — works end-to-end against the real bootstrapped repo:
real PAT auth, real push, real GitHub Pages serving. Close the one unit-test
gap (untested default `_remote_url` construction) alongside it.

## Non-goals

- Re-testing rotation/size-check/redaction live — already thoroughly unit
  tested with a local bare repo; a live re-run of the same logic against a
  real remote adds risk (more real pushes) without adding signal.
- Testing `publish()` and `update_feed()` as independent calls — production
  code only ever calls the atomic combined method.
- Building gh-pages branch bootstrapping into the publisher itself — still
  explicitly out of scope per the module's own docstring; this session's
  bootstrap was a one-time manual setup step for the test repo, same as the
  docstring describes for any real deployment.

## Approach

### Unit test addition

One new test in `tests/test_publishers.py`, alongside the existing
`GitHubPagesPublisher` tests: construct a publisher without passing
`remote_url`, assert `publisher._remote_url == f"https://{token}@github.com/{repo}.git"`
for a fixed fake `repo`/`token` pair. No network, no git calls — pure
construction-logic check.

### Live verification

1. Build a `GitHubPagesPublisher` the same way `core/pipeline.py`'s
   `_build_publisher` does: `repo="easilyBaffled/reader-rss-source"`,
   `token=` the real `GITHUB_PAT` from `.env` (loaded via the existing
   `load_config()` / `python-dotenv` mechanism already used for `KOKORO_API_KEY`
   etc. — not hand-rolled), a throwaway local `work_dir`, default
   `branch="gh-pages"`, a `FeedConfig`. No `remote_url` override — this is
   the one path the new unit test checks in isolation, and the live run
   confirms it actually works against the real host.
2. Construct one fake `Episode` + a small fake mp3 file (reuse the same
   `_episode()` / fake-bytes pattern from `test_publishers.py` rather than
   inventing a new fixture style).
3. Call `publish_and_update_feed(episode, audio_path, [episode])`, capture
   the returned `(public_url, feed_url)`.
4. Confirm via the GitHub API or `git ls-remote` that the push landed on the
   real `gh-pages` branch (fast, doesn't depend on Pages' propagation delay).
5. Poll the returned URLs with `curl` until they 200 (GitHub Pages takes up to
   ~1 minute to serve a freshly pushed branch, per the module's own comment) —
   confirm the mp3 bytes and feed.xml content match what was pushed.
6. If anything fails: root-cause with `systematic-debugging` (read the actual
   git/HTTP error, don't guess), and decide fix-now vs defer based on whether
   it's a real code bug (fix it, write a regression test, same bar as this
   session's other fixes) vs. an environmental issue (e.g. token scope
   wrong, Pages not enabled on the repo yet — note as a setup gap instead).

## Success criteria

`publish_and_update_feed()` against the real repo returns URLs that are
fetchable and serve the exact pushed content. The new `_remote_url`
construction unit test passes. Full `uv run pytest` suite stays green.

## Testing

- New unit test: `tests/test_publishers.py`, construction-only, no network.
- Live run: manual, via a one-off script/REPL call (not added to the
  automated suite — it requires a real PAT and real network, same reasoning
  `tests/test_publishers.py`'s existing bare-repo approach avoids hitting a
  real GitHub remote in CI).
